# Architecture comparison — TLS / GC pixel-segmentation on TCGA WSI

This is the head-to-head comparison of the **most successful TLS / GC
segmentation architectures** tried on the TCGA-BLCA + TCGA-KIRC +
TCGA-LUSC cohort (170 test patients excluded from all training; 5-fold
patient-stratified CV on the remaining cohort, seed=42).

Per-architecture training files and run scripts live under
`recovered/scaffold/`. All numerical artefacts and the figures below
are generated by `notebooks/build_arch_comparison.py` from the
on-disk per-fold JSONs and the master `benchmark_5fold.csv`.

> **Detailed companion:** `ARCHITECTURE_DETAILS.md` — full cohort
> definition, TLS/GC distribution per cancer, per-architecture module
> breakdown + param counts, and complete training hyperparameters
> (loss, optimizer, LR schedule, epochs, batch, AMP, augmentation,
> seed) for each model.

## Training supervision

| Architecture | Loss | Per-pixel/per-patch target | Class weighting / extras | Saw GT-negative slides during training? |
|---|---|---|---|---|
| **Cascade Stage 1** (GraphTLSDetector) | `BCEWithLogitsLoss(pos_weight)` per UNI patch (1 logit / patch). | Patch label from upsampled mask via `patch_labels_from_mask`: `1` if any positive pixel in the patch, else `0`. | `pos_weight` to balance the heavy negative skew of the patch graph. | **YES** — `TLSSegmentationDataset` admits all 1015 slides; missing-mask entries get an all-zero target (`prepare_segmentation.py:188`). |
| **Cascade Stage 2** (RegionDecoder) | Weighted CE + GC-soft-Dice: `CE(weights=[1,5,3]) + gc_dice_weight × (1 − soft_Dice_GC)`. | 3-class per-pixel targets {bg, TLS, GC} on 768×768 region windows; `INVALID_MASK_VALUE=255` ignored. | Class weights `[1.0, 5.0, 3.0]`, `gc_dice_weight=1.0` (boosts the rare GC class so it doesn't get drowned by CE). | **NO** — `tls_patch_dataset.py:106` only collects "TLS-positive patches". Stage 2 only ever sees 3×3 windows around real TLS pixels. *However*, Stage 1 is the gate — on a true-negative slide Stage 1 should select nothing → Stage 2 never runs → cascade resolves as TN by construction. |
| **GNCAF v3.56 / v3.58 / v3.59** | Weighted CE + soft-Dice (mean over all 3 classes): `CE(weights=[1,5,3]) + dice_loss_weight × soft_Dice`. | 3-class per-pixel targets on 256×256 patches. | Class weights `[1.0, 5.0, 3.0]`, `dice_loss_weight=0.5`. | **YES** — every config sets `include_negative_slides: true, neg_slide_targets: 8`; `gncaf_dataset.py:160-165` admits negatives when the flag is on. |

**Implications for the eval comparison:**

* The cascade is supervised at *two granularities* (patch-level binary
  for Stage 1, pixel-level 3-class for Stage 2). Stage 1's "saw negatives"
  exposure is what protects the cascade from triggering Stage 2 on
  GT-negative slides.
* GNCAFs are one big pixel-level 3-class problem on every patch — the
  `neg_slide_targets: 8` knob mixes 8 patches per negative slide into
  each batch as a regulariser.
* Reported `val_mDice` during GNCAF training is mean Dice over {bg,
  TLS, GC} on validation patches. Reported `val_mDice` during Stage 2
  training is mean Dice over only Stage-1-positive 3×3 windows. They
  are **not directly comparable** to each other — the cascade Stage 2
  number is on a much harder subset (the eval pixel-Dice in the table
  below is the apples-to-apples comparison).

## Per-slide instance counts vs. metadata gold standard

For each slide we compare **predicted** instance count (8-conn CCs of
the predicted patch grid for each class, with 1-iter binary closing)
against the **gold-standard metadata** count (`gt_n_tls`, `gt_n_gc`)
from the HookNet annotation pipeline.

|                | TLS sp | TLS pe | TLS R² | TLS MAE | GC sp | GC pe | GC R² | GC MAE |
|----------------|--------|--------|--------|---------|-------|-------|-------|--------|
| **Cascade v3.37** | **0.891** | **0.874** | **0.742** | **5.43** | **0.933** | **0.969** | **0.888** | **0.35** |
| GNCAF v3.56    | 0.662  | 0.594  | −1.74  | 19.19   | 0.618 | 0.758 | 0.553 | 0.96   |
| GNCAF v3.58    | 0.576  | 0.504  | −4.82  | 32.48   | 0.569 | 0.727 | 0.478 | 1.07   |
| GNCAF v3.59    | 0.528  | 0.387  | −9.10  | 46.47   | 0.642 | 0.589 | 0.305 | 1.00   |

**Headline:** the cascade is the only architecture whose per-slide count
predictions explain the gold-standard variance (TLS R² = 0.742, GC R² = 0.888).
The GNCAFs all have negative TLS R² because their pixel decoders
systematically over-segment small TLS clusters into many tiny components
(11+ predicted CCs even for slides with 1–3 GT TLS). v3.59's R² is the
lowest of all (concat-all-hops actively makes the over-segmentation worse).
The cascade does much better on counts because RegionDecoder operates on
tighter ROIs and produces cleaner contiguous instance masks.

## Bucketed evaluation — is the predicted instance count in the right bin?

We bin slides by their gold-standard count and ask: how often does the
predicted count fall in the *same* bin? This is a strict per-bin
classification (not just correlation).

* **TLS bins:** {0, 1–3, 4–10, 11+}. The cohort has no TLS-zero slides
  in the GNCAF-eval set (all val slides have ≥1 TLS); for "11+" slides
  recall is high but precision low because the GNCAFs over-predict.
* **GC bins:** {0, 1, 2–3, 4+}. Most slides are GC-zero; on this
  largest bin both v3.56 and v3.58 reach ≥0.80 P/R/F1.

## Slide-level detection — AUROC and PR-AUC

These are AUROC and Average Precision (PR-AUC) for "any-instance"
binary detection per slide, using the **predicted instance count
`n_pred` as the ranking score**. We don't have a per-slide
probabilistic detection score on disk (the eval pipeline thresholds
patches at 0.5 then counts CCs), so the curves slide over count
thresholds (≥1, ≥2, …). This is an honest read on whether the model's
count *ranking* separates GT-positive from GT-negative slides.

| | TLS AUROC | TLS PR-AUC | GC AUROC | GC PR-AUC | GC n_pos / n_neg |
|---|---|---|---|---|---|
| **Cascade v3.37** | n/a (TLS+) | 1.00 | **0.979** | **0.954** | 39 / 85 |
| GNCAF v3.56  | n/a (TLS+) | 1.00 | 0.781 | 0.668 | 168 / 306 |
| GNCAF v3.58  | n/a (TLS+) | 1.00 | 0.750 | 0.632 | 162 / 284 |
| GNCAF v3.59  | n/a (TLS+) | 1.00 | 0.780 | 0.628 | 39 / 85 |

* **TLS:** every val slide in this cohort has ≥1 GT TLS, so AUROC is
  undefined and PR-AUC trivially equals the (100 %) positive
  prevalence. The TLS PR-AUC column is shown only for completeness.
* **GC:** the cascade leads decisively (AUROC 0.979 vs ~0.75–0.78 for
  GNCAFs; PR-AUC 0.954 vs ~0.63–0.67). Among the GNCAFs, v3.59
  (rigour fix) and v3.56 are essentially tied on AUROC (~0.78); v3.58
  (the augmentation variant) is slightly behind (0.75).
* The GC AUROC ranking matches what the detection F1s above already
  showed, but the PR-AUC numbers expose a much wider quality gap on
  the precision axis: cascade can sweep through high-recall
  thresholds without losing precision, while the GNCAFs collapse to
  ~50 % precision at high recall.

## Notes / caveats

1. **Cascade per-slide rows are now on disk** (`cascade_per_slide.json`,
   keyed by threshold). The patch landed in `eval_gars_cascade.py` and
   the v3.37 fold-0 4-shard re-eval was the source for the cascade
   columns above.
2. **R² values are computed against y=GT** (not against y=mean(GT)),
   so the numbers measure *how much of the GT variance the prediction
   explains*. Negative R² → the model's prediction is worse than just
   predicting the GT mean — the over-prediction noise on tiny TLS sets
   blows R² up especially badly.
3. **Bucketed F1 = 0** when both precision and recall are 0 (vs NaN);
   shown so the bar plot is consistent.
4. **GNCAF v3.58 fold-2 was an outlier** (val mDice ≈ 0.27, dragging
   the 5-fold mean below v3.56). The same hyperparams without the
   augmentation flag (v3.56) are stable on fold 2.
